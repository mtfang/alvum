# Minimal Viable Path to PMF

A scope-cut plan that tests the core alvum thesis with the smallest possible artifact, before any infrastructure that can't pay off until the thesis is proven. This is adversarial to every comprehensive plan in this repo. It is the right plan.

## The Thesis in One Sentence

**Trailing decision graph → useful morning briefing → behavior change.**

Three layers, tested in order:

1. **Layer 1 (the thesis)** — does a briefing derived from an automatic record of yesterday produce something a user wants to read daily?
2. **Layer 2 (alignment)** — does intention tracking sharpen the briefing into actionable gap reports?
3. **Layer 3 (agency)** — does proactive in-the-moment prompting drive behavior change?

Layer 1 is a lower bar and a higher-leverage test. If Layer 1 fails — if the briefing is LLM-slop, or users don't read it, or reading it changes nothing — Layers 2 and 3 are moot. **Everything in this document tests Layer 1 only.** Layer 2 and 3 are explicitly out of scope until Layer 1 is proven.

## What "Proven" Means

PMF signals, in order of strength:

1. **Retention**: after 14 days, the user still opens the briefing daily.
2. **Action attribution**: user, unprompted, cites the briefing in a decision ("I saw in this morning's briefing that..." / "I closed the migration thread because alvum flagged it as stale").
3. **Referral**: user tells another person about alvum without being asked.
4. **Payment willingness**: user says yes to "would you pay $X/mo for this?" (don't actually charge yet; the question is the signal).

Target for go-ahead on Layer 2: **5 of 7 users hit signal #1 + at least 3 of them hit signal #2 + at least 1 hits #3 within 14 days.**

Target for kill / pivot: **fewer than 3 hit #1, OR zero hit #2 within 14 days.** The briefing content is either wrong or the core thesis is wrong. Either way, stop building.

## Current State (don't rebuild what's done)

Already shipped and validated against real data:
- ✅ Claude Code connector (sessions → observations)
- ✅ Decision extractor (observations → Decisions with actor attribution)
- ✅ Causal linker (Decisions → graph with mechanism + strength)
- ✅ Briefing generator (graph → markdown briefing)
- ✅ CLI wired end-to-end: `alvum extract --session <path> --output ./output`
- ✅ Output validated: 40-41 decisions + 73-74 causal links on a single design session

**The V0 artifact IS the PMF candidate.** Not hypothetically — literally. The question isn't *"what should we build to test PMF?"*, it's *"does what we already built pass the bar?"*

Everything below is staged so that the first test uses ONLY what's already built, and each subsequent step only ships if the prior step's signal warranted it.

## Stage 0: Dogfood the Existing Artifact (Week 1)

**Before extending anything**, prove the existing artifact is at least interesting.

- Commit to running `alvum extract` on yesterday's Claude Code sessions every morning for 7 consecutive days. Cron it: `0 7 * * * cd ~/git/alvum && ./run-daily-briefing.sh`.
- Read the resulting briefing within 30 minutes of waking.
- Keep a two-line journal: (a) did I actually open it? (b) did anything in it drive an action or a thought I wouldn't have had otherwise?

Running cost: zero. Infrastructure built: zero.

**Stage 0 exit criterion:** after 7 days, can you honestly say the briefing earned 3 minutes of attention on ≥ 5 of 7 days AND produced at least one action you attribute to it?

- **Yes** → Proceed to Stage 1. Briefing content is promising; widen the data.
- **No** → Do not extend. Iterate on the **briefing prompt** (in `alvum-pipeline/src/briefing.rs`) for one more week. If Stage 0 still fails after prompt iteration, **the thesis is in trouble.** The problem is either: (a) your Claude Code sessions aren't decision-rich enough in isolation (move to Stage 1 to test with wider data) or (b) the briefing synthesis is fundamentally LLM-slop-shaped and no amount of data will save it. Honestly assess which.

## Stage 1: Add the Single Highest-Value Data Source (Week 2)

If Stage 0 passes, widen capture to include the one source most likely to densify decision signal per hour of user time.

**Choice: screen capture only.** Not audio. Not location. Not health. Not wearable. Not CarPlay. Just macOS screen events (app focus + window titles + a11y diffs → `events.jsonl`).

Why screen first:
- Zero hardware friction (runs on the Mac you already have).
- Zero privacy-inspection overhead compared to audio (no transcription of other voices, no ambient captures of your partner / coworkers).
- Decision density is high for knowledge workers (your persona).
- Infrastructure is already ~70% built in `alvum-capture-screen` and `alvum-connector-screen`.

What to NOT add yet:
- Audio (Whisper latency + transcription quality + room-mic issues + coworker privacy = expensive to get right).
- Wearable (hardware, firmware, pairing, mDNS — weeks of work per the wearable spec).
- Location / health / Strava / HealthKit (all deferred).

**Scope for Stage 1:**
- Finish the screen-capture daemon: app-focus events, window-title events, and basic a11y text diffs. That's it. No clipboard, no browser URL tracking, no semantic event classification. Just the five simplest detection patterns described in the spec's § Semantic Change Events.
- Wire screen events into the existing pipeline as additional Observations.
- Re-run the briefing — does it get sharper now that it has *both* the code-side decisions (Claude Code) and the non-code activity (screen)?

**Stage 1 exit criterion:** same as Stage 0 but over 7 new days (days 8-14). Briefing opened on ≥ 5 of 7 days AND produced ≥ 2 attributed actions across the week.

- **Yes** → Proceed to Stage 2. Expand beyond N=1.
- **No** → Same diagnosis as Stage 0's "no" branch. Either iterate on the briefing prompt using richer data, or confront the possibility that the thesis is wrong.

## Stage 2: Expand to N=5-7 Trusted Users (Weeks 3-5)

If Stages 0 and 1 both pass for the builder (N=1), test with the right users. Do not go wide. Go with 5-7 close-fit people.

**Target persona (tight):**
- AI-savvy engineers, knowledge workers, or designers who heavily use Claude Code / Cursor / similar.
- macOS-only (alvum is macOS-only per the spec).
- Willing to talk to you for 30 min at day 7 and day 14.
- Explicit: not your parents, not your random friends from high school, not "anyone who'd try it."

Why tight persona: if the briefing doesn't work for this persona, it won't work for anyone. If it does work, the persona expansion (to non-engineers, different OS, etc.) is a *later* exercise, and you'll have a believable foundation.

**What they install:**
- A simple bash-one-liner that: clones the repo, sets up the Claude CLI provider, runs `alvum capture` as a launchd daemon, schedules `alvum extract` at 3am daily, and drops briefings into `~/.alvum/generated/briefings/YYYY-MM-DD/briefing.md`.
- Optional: a tiny macOS Shortcut that emails the briefing to the user's own email at 7am — zero UI to build; user reads in Mail app at breakfast.

**Explicit skip list for this stage:**
- No Electron app.
- No web UI.
- No menubar widget.
- No intentions page (no intentions at all in Stage 2 — see below).
- No evening check-in (one-way information flow only: alvum → user).
- No decisions-graph visualization.
- No settings page; everything is in `~/.config/alvum/config.toml`.
- No onboarding flow beyond the install script.

**Explicitly NOT in the briefing at Stage 2:**
- **Intentions / alignment / drift language.** The briefing is descriptive, not prescriptive. "Here's what happened. Here's what's connected. Here are open threads." No "you're drifting from goal X." Intentions require setup friction (user must define goals) and sharper quality bars; defer until Stage 3.
- **Behavioral signals.** Aborted actions and avoidance patterns need screen + audio fusion; out of scope.
- **Cross-domain butterfly effects.** Spec's most ambitious briefing feature; requires rich multi-week data. Not testable at N=7/14days.

The briefing at Stage 2 is deliberately minimal. Three sections:

1. **What happened yesterday.** 3-5 paragraphs narrating the shape of the day, grounded in Claude Code sessions + screen activity.
2. **Decisions made.** Bulleted list of decisions extracted, with actor attribution and causal chains to prior decisions.
3. **Open threads.** Decisions from prior days whose outcomes haven't been observed yet. Just a list; no scoring.

That's it. No alignment section. No check-in. No scoring. No predictions.

**Daily use protocol for each user:**
- Briefing lands at 7am in their email.
- They read it at breakfast.
- You (the builder) interview each user at day 7 and day 14 over 30 min each: *"Did you open it? Did you act on anything? Would you be upset if it disappeared?"*

**Stage 2 exit criterion (the actual PMF test):**
- ≥ 5 of 7 users read the briefing on ≥ 10 of 14 days.
- ≥ 3 of 7 users cite a specific decision/action they attribute to the briefing.
- ≥ 1 of 7 users tells another person about alvum without being prompted.

**Hit all three:** the core thesis is validated. Proceed to Stage 3 (add intentions + alignment — Phase A of the alignment-primitives plan, and Phase B of the alignment engine).

**Hit #1 but not #2:** the briefing is entertaining but not actionable. Iterate on the content, not the infrastructure. Specifically: what does the briefing say that could be said differently to drive action? Revisit Stage 2 with prompt changes.

**Miss #1:** the briefing isn't worth reading. Either content is wrong or the thesis is wrong. Do not build Stage 3. Hard conversation about whether to continue.

## Stage 3: First Honest Alignment Test (if Stage 2 passes)

If Stage 2 hits all three signals, *then* add intentions and an alignment section to the briefing. This is where the comprehensive Phase A and Phase B plans in this repo become the right work — but only after Stage 2 validated that there's something to align against.

Stage 3 scope (for reference only; detailed plan deferred until Stage 2 green):
- Ship the Phase A alignment-primitives types (Decision outcome tracking, Intention, Evidence, EmergentState, BehavioralSignal, AlignmentReport).
- Onboarding dialogue adds domain selection + initial goal.
- Briefing gains an "Alignment" section: for each active intention, status + evidence + trend.
- Evening check-in — 2-3 questions delivered as a markdown file the user edits and saves; pipeline picks up the edits overnight.

New PMF bar for Stage 3: existing users still read daily, AND at least 3 of 7 describe a concrete change in a tracked habit or goal over a 30-day window.

Stage 3 is where Layer 2 of the thesis gets tested. Don't fight this battle before winning Layer 1.

## Explicitly Out of Scope Until After Stage 3 Passes

All of the following are in the other plans/specs in this repo and should be treated as **write-only until Layer 2 is validated**:

- Electron app shell, menubar widget, iOS companion, watchOS companion
- Device fleet (pairing, heartbeats, registry)
- Location / map, multi-source fusion, Strava, HealthKit routes
- Health connector (HR, sleep, workouts, HRV)
- Wearable ESP32 firmware
- CarPlay bridge
- Managed tier / encrypted cloud sync
- Agency Layer (Now, Find, Quiet zones, Life phase)
- Text / multimodal embeddings
- Fine-tuning pipeline, RLHF reward modeling
- Decision graph visualization
- `/intentions`, `/decisions`, `/knowledge`, `/devices`, `/connectors`, `/timeline` web routes
- Multi-device / multi-Box

Each of these is a real feature worth building **later**. Any effort spent on them before Stage 2 validates is effort likely to be thrown away if Layer 1 fails, and effort that delays Layer 1 validation if Layer 1 succeeds.

## The Hard Skip: Proactive Agency Until Layer 2 Proves Out

The `2026-04-18-agency-layer.md` spec I wrote earlier — Now, Find, Quiet zones, Life phase — is strategically the product's highest-leverage bundle. It is also **at least six months away** from being the right thing to build. A proactive "what should I do right now?" surface has no foundation to build on if:
- The briefing isn't trusted (Stage 1-2 not validated)
- There are no confirmed intentions to align against (Stage 3 not shipped)
- There's no long-enough decision graph to make absence detection meaningful (< 3 months of data)

Discipline: the Agency Layer spec stays on the shelf until Stage 3 is live and has been used for a month.

## What Needs To Be Built (minimized)

For Stage 1:
- Finish the screen-capture a11y differ (scope: the 5 simplest patterns — app focus change, window title change, labeled-value change, text-content change, node appeared/disappeared). Defer clipboard + URL triggers.
- Wire `alvum-connector-screen` output into the existing pipeline as Observations.

For Stage 2:
- Write the install bash script.
- Write the Mac Shortcut that emails the latest `~/.alvum/generated/briefings/*/briefing.md` daily.
- Write a simple `alvum status` CLI that shows "last extract succeeded at X, last briefing at Y" — for user debugging.

Estimate: **~2 engineer-weeks of work total** to reach the Stage 2 entry point. Everything else in the comprehensive plans is deferred.

## The One Metric That Matters Through Stage 2

**Daily briefing open rate per user.**

Not time-in-app. Not decisions extracted. Not latency. Not any of the vanity metrics. The only question that matters is: did the user look at what alvum produced today?

Instrument this via a tiny pixel-tracking thing in the briefing email, or more crudely, `stat -c %X ~/.alvum/generated/briefings/*/briefing.md` on each user's machine for file-access time. Any technique that answers "did they open it" is sufficient.

## Kill Criteria (explicit)

Stop building and hold a hard conversation if any of:

1. **Stage 0 fails after prompt iteration.** The briefing isn't useful for the builder, even with existing rich Claude Code data.
2. **Stage 1 fails.** Adding screen data didn't sharpen the briefing enough for daily use.
3. **Stage 2 misses open-rate signal.** < 3 of 7 users read the briefing ≥ 10 of 14 days.

Failure at any of these is not embarrassing. It's the system working. A plan that can't fail isn't a test.

## What Success Looks Like End of Week 6

The short version: a plain-text tool running as a launchd daemon on 7 Macs that emails 7 people a decision-graph-grounded briefing every morning, and at least 5 of them open it daily, and at least 3 of them attribute a real action to it, and at least 1 has told someone else about it.

That is the signal. That is what everything else — Electron, wearable, health, location, managed tier, agency — is scaled from. Nothing earlier is worth building at scale.

If we get there, the rest of the roadmap is a scaling exercise. If we don't get there, we need to be honest about why before building anything else.
